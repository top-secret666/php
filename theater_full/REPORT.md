6. Настроить аутентификацию и доступ.

**Объект исследования:** процессы продажи и учета билетов в театре.

## Анализ предметной области

Театральная касса — это система, которая поддерживает полный цикл работы с показами и билетами:
- **Площадки (venues)** — описание театров и залов.
- **Постановки (shows)** — набор спектаклей с описанием и характеристиками (длительность, возрастной рейтинг, язык).
- **Сеансы (performances)** — конкретные показы спектаклей в определенную дату/время.
- **Секции и места (seat_sections, seats)** — разбиение зала на секции и конкретные места, включая тип места и активность.
- **Ценовые уровни (price_tiers)** — стоимость билетов, заданная в центах и привязанная к площадке/секции/постановке.
- **Заказы (orders)** — агрегируют покупку одного или нескольких билетов.
- **Билеты (tickets)** — конкретные единицы продажи с QR‑кодом, статусом и временем выдачи.
- **Платежи (payments)** — учет финансовых транзакций.

Особенности предметной области:
- Билет уникален для пары «сеанс — место».
- Один заказ может включать несколько билетов.
- Цена может зависеть от секции и/или постановки.
- Билет может быть «зарезервирован», «продан» либо «отменен».

### Типовые сценарии (use‑cases)

1. **Просмотр афиши** — гость открывает страницу списка постановок, фильтрует по названию, выбирает спектакль.
2. **Выбор сеанса** — пользователь открывает карточку спектакля и выбирает дату/время сеанса.
3. **Бронирование места** — создается билет со статусом `reserved` для пары «сеанс‑место».
4. **Оформление заказа** — пользователь создает заказ, указывает сумму, связывает билеты с заказом.
5. **Оплата и выдача** — билет переводится в статус `sold`, генерируется QR‑код.
6. **Проверка билета** — при входе билет отмечается временем `checked_in_at`.

### Нефункциональные требования

- **Надежность:** предотвращение двойной продажи одного места (уникальный индекс на `performance_id` + `seat_id`).
- **Производительность:** индексирование внешних ключей, пагинация списков (по 15–30 записей).
- **Безопасность:** валидация входных данных, защита маршрутов через `auth`.
- **Сопровождаемость:** единая точка валидации через FormRequest; покрытие тестами ключевых сценариев.

### Функциональные требования

1. Управление постановками: создание, просмотр, редактирование, удаление.
2. Управление сеансами: создание сеансов для постановок.
3. Управление билетами: резервирование, продажа, отмена.
4. Оформление заказов и привязка билетов к заказу.
5. Генерация QR‑кода при продаже билета.
6. Проверка (check‑in) билета на входе.
7. Регистрация и аутентификация пользователей.

### Ограничения и допущения

- Интеграция с реальными платежными провайдерами не реализована (модель данных предусмотрена).
- Сквозная аналитика по продажам вынесена в будущие улучшения.
- В рамках лабораторной работы допускается упрощенная роль администратора.

## Проектирование базы данных (ER‑диаграмма)

В проекте подготовлены схемы:

**Ключевые сущности и связи:**

- `performances 1..* tickets` — у сеанса множество билетов.
- `orders 1..* tickets` — заказ включает билеты.

## Описание архитектуры приложения

Приложение реализовано на Laravel и придерживается архитектуры MVC:
- **Model** — Eloquent‑модели, отражающие сущности предметной области.
- **View** — Blade‑шаблоны для рендеринга списков и форм.
- **Controller** — контроллеры для обработки CRUD‑операций.

**Слои:**
- Маршрутизация (`routes/web.php`) — REST‑маршруты для сущностей.
- Бизнес‑логика — частично реализована в контроллерах, часть валидации вынесена в FormRequest.
- Доступ — базовая авторизация через middleware `auth`.

**Технологии:**
- PHP 8.2+  
- Laravel 12.x  
- SQLite для тестов (in‑memory)  
- PHPUnit для тестирования  

### Структура проекта (кратко)

- `app/Models` — модели Eloquent
- `app/Http/Controllers` — контроллеры
- `app/Http/Requests` — FormRequest‑валидация
- `resources/views` — Blade‑шаблоны
- `routes/web.php` — маршрутизация

### Поток обработки запроса

1. Запрос приходит по маршруту `web.php`.
2. Контроллер получает объект `Request`/`FormRequest`.
3. Валидация выполняется автоматически в FormRequest.
4. Контроллер вызывает модель, формирует ответ/редирект.
5. Ответ возвращается пользователю (Blade view или редирект).

## Алгоритмы и бизнес‑логика

### Алгоритм резервирования билета
1. Пользователь выбирает сеанс и место.
2. Система проверяет уникальность пары `performance_id` + `seat_id`.
3. Создается запись в `tickets` со статусом `reserved`.

### Алгоритм покупки билета
1. Создается заказ (`orders`) с суммой `total_amount`.
2. Билет связывается с заказом (`order_id`).
3. При смене статуса на `sold` генерируется QR‑код.

### Алгоритм check‑in (проверка билета)
1. Контроллер принимает запрос на обновление билета.
2. Валидация включает дату `checked_in_at`.
3. Время прохода записывается в поле `checked_in_at`.

### Обработка цен
Цена хранится в `price_tiers.amount_cents`. При создании билета она переводится в дробный формат:
`price = amount_cents / 100`.

## Реализация основных компонентов

### Модели и отношения

Ключевые модели:
- `Show`, `Performance`, `Ticket`, `Order`, `Seat`, `SeatSection`, `Venue`, `PriceTier`, `Payment`.

Пример отношений:
- `Show` → `Venue` (belongsTo)
- `Performance` → `Show` (belongsTo)
- `Ticket` → `Performance`, `Seat`, `Order` (belongsTo)

Дополнительные связи:
- `SeatSection` → `Venue` (belongsTo)
- `Seat` → `SeatSection` (belongsTo)
- `Order` → `User` (belongsTo)
- `Ticket` → `User` (purchaser)

**Фрагмент (пример):**
```php
class Ticket extends Model
{
    protected $fillable = ['performance_id','seat_id','order_id','purchaser_id','price','status','qr_code','issued_at','checked_in_at'];

    public function performance() { return $this->belongsTo(Performance::class); }
    public function seat() { return $this->belongsTo(Seat::class); }
    public function order() { return $this->belongsTo(Order::class); }
}
```

### Валидация данных

Валидация реализована через FormRequest классы:
- `StoreShowRequest`
- `StoreOrderRequest`
- `StoreTicketRequest`

Пример:
```php
public function rules(): array
{
    return [
        'user_id' => 'required|exists:users,id',
        'total_amount' => 'required|numeric|min:0',
        'status' => 'nullable|string',
    ];
}
```

**Причина выбора FormRequest:**
- единообразие правил;
- упрощение контроллеров;
- более простое тестирование правил.

- `LoginController`, `RegisterController`.
- Middleware `auth` ограничивает доступ к созданию и редактированию сущностей.

Права администратора могут быть расширены через `is_admin` в `users` и отдельные middleware.

## Листинг ключевых фрагментов кода

**Генерация QR‑кода для билета при продаже:**
```php
if (isset($data['status']) && $data['status'] === 'sold' && empty($data['qr_code'])) {
    $data['qr_code'] = bin2hex(random_bytes(8));
}
```

**Привязка цены к ценовому уровню:**
```php
$tier = PriceTier::find($data['price_tier_id']);
$data['price'] = $tier->amount_cents / 100;
```

**Проверка билета:**
```php
'ticket_id' => 'required|exists:tickets,id'
```

**Обновление времени проверки билета:**
```php
'checked_in_at' => 'nullable|date'
```

**Переход билета в статус «продан»:**
```php
$ticket->update([
    'status' => 'sold',
    'order_id' => $order->id,
]);
```

**Пример теста проверки билета:**

### Поиск, фильтрация, сортировка и пагинация (афиша)

В приложении реализован список спектаклей (афиша) с обработкой параметров запроса.

**Маршруты:**
- `GET /shows` — список спектаклей;
- `GET /shows/search` — тот же список, но используется как endpoint для формы поиска.

**Параметры фильтрации и сортировки:**
- `q` — строка поиска по полям: `title`, `description`, `director`;
- `director` — точный фильтр по режиссёру;
- `sort` — сортировка (`title`, `duration_minutes`, `created_at`);
- `dir` — направление сортировки (`asc`/`desc`).

**Техническая реализация:**
- логика находится в `ShowController::getFilteredShows()`;
- используется `paginate(15)` и `appends($request->query())`, чтобы параметры сохранялись при переходе по страницам;
- поиск выполнен через `like` (совместимо с SQLite/MySQL/PostgreSQL), чтобы не зависеть от `ilike`.

### Загрузка файлов (постер спектакля)

Для предметной области театра логично хранить постер спектакля. В проекте реализована загрузка изображения с сохранением в файловое хранилище.

**Где реализовано:**
- формы: `resources/views/shows/create.blade.php`, `resources/views/shows/edit.blade.php` (`enctype="multipart/form-data"`, поле `poster`);
- валидация: `app/Http/Requests/StoreShowRequest.php` (`poster` — `image|max:4096`);
- обработка: `app/Http/Controllers/ShowController.php` (сохранение на диске `public`, запись URL в поле `poster_url`).

**Примечание:** для раздачи файлов используется `Storage::url(...)`, а в production обычно настраивается симлинк `storage -> public/storage`.

## CI/CD и автоматизация

В проекте используется GitHub Actions (workflow: `.github/workflows/ci.yml`) для автоматической проверки качества.

**Ключевые шаги CI:**
- запуск job в каталоге `theater_full/` (там расположен `composer.json` и `artisan`);
- установка зависимостей через Composer;
- подготовка окружения: копирование `.env.example` → `.env`, создание SQLite файла `database/database.sqlite`;
- генерация ключа приложения (`php artisan key:generate`);
- миграции и сиды (`php artisan migrate:fresh --seed --force`);
- запуск тестов (`php artisan test`).

**Почему это важно:**
- исключает ситуацию, когда CI запускается не из директории Laravel‑проекта и не находит `composer.json`;
- обеспечивает воспроизводимость проверок на чистой базе данных (SQLite), что соответствует требованиям задания.
